# Архитектурный план реализации временных ТМЦ в Django клиенте

## Контекст
Временные ТМЦ уже реализованы на стороне SyncServer. Django клиент `Warehouse_web` должен быть доработан для поддержки полного цикла работы с временными ТМЦ:
- Создание операций с временными ТМЦ (вместе с постоянными)
- Модерация временных ТМЦ (список, карточка, approve/merge)
- Корректное историческое отображение операций после approve/merge
- Интеграция с существующими компонентами (SyncServerClient, sidebar, права доступа)

## Ключевые ограничения
- Не создавать локальную Django‑модель `TemporaryItem`
- Не кэшировать временные ТМЦ в `catalog_cache`
- Не подменять исторические snapshot‑поля текущими данными каталога
- Модерация доступна только пользователям с `can_manage_catalog` (root, chief_storekeeper)

## Модули реализации

### 1. Транспорт и контракты (`temporary_items_api.py`)
**Цель**: предоставить Python‑клиент для работы с endpoint‑ами временных ТМЦ SyncServer.

**Файлы**:
- `apps/sync_client/temporary_items_api.py` (новый)
- `apps/sync_client/__init__.py` (добавить импорт)
- `apps/sync_client/exceptions.py` (используются существующие)

**Методы API** (аналогично `catalog_api.py`):
- `list_temporary_items(filters=None, *, acting_user_id=None, acting_site_id=None)`
- `get_temporary_item(temporary_item_id, *, acting_user_id=None, acting_site_id=None)`
- `list_temporary_item_operations(temporary_item_id, filters=None, *, acting_user_id=None, acting_site_id=None)`
- `approve_as_item(temporary_item_id, *, acting_user_id=None, acting_site_id=None)`
- `merge_to_item(temporary_item_id, payload, *, acting_user_id=None, acting_site_id=None)`

**Зависимости**:
- `SyncServerClient` (базовый клиент)
- Существующие `SyncServerAPIError`

**Тесты**: unit‑тесты на все методы, проверка обработки ошибок.

### 2. Создание операций с временными ТМЦ
**Цель**: переработать форму создания операции, чтобы поддерживать смешанные строки (каталожные и временные), генерировать `client_request_id` и отправлять корректный payload.

**Изменяемые файлы**:
- `apps/operations/views.py` – функция `_build_create_payload()` и класс `OperationCreateView`
- `apps/operations/forms.py` – возможно, расширение `OperationCreateForm`
- `templates/operations/form.html` – JS‑логика для управления черновиком
- `templates/operations/form.html` – UI для добавления временной строки

**Основные изменения**:

#### 2.1. Расширение формата черновика
Черновик операции должен хранить строки с признаком `kind`:
```json
{
  "operation_type": "RECEIVE",
  "site_id": 1,
  "client_request_id": "7e31b0ee-f7b3-4fb6-b7a8-4d31ab6d89fb",
  "items": [
    {
      "kind": "catalog",
      "item_id": 101,
      "name": "Кабель ВВГнг 3х2.5",
      "sku": "VVG-325",
      "unit_symbol": "м",
      "quantity": "5"
    },
    {
      "kind": "temporary",
      "client_key": "tmp-2",
      "name": "Кабель без карточки",
      "sku": "",
      "unit_id": 3,
      "unit_symbol": "м",
      "category_id": 7,
      "category_name": "Кабели",
      "description": "",
      "quantity": "2"
    }
  ]
}
```

#### 2.2. Генерация `client_request_id`
- Если в черновике есть хотя бы одна временная строка, необходимо сгенерировать UUID v4 и сохранить его в черновике.
- При повторной отправке того же черновика использовать тот же `client_request_id`.
- `client_request_id` передаётся в payload операции на уровне операции (не строк).

#### 2.3. Адаптация `_build_create_payload()`
- Преобразовывать строки с `kind: "catalog"` в `item_id`.
- Преобразовывать строки с `kind: "temporary"` в объект `temporary_item` с полями `client_key`, `name`, `sku`, `unit_id`, `category_id`, `description`.
- Запрещать склеивание временных строк по имени (каждая временная строка – отдельная сущность).
- Проверять, что для временных строк заполнены обязательные поля (name, unit_id, category_id).

#### 2.4. UI‑изменения в форме операции
- Добавить кнопку/блок «Добавить временную ТМЦ».
- При выборе временной ТМЦ показывать поля: наименование, SKU (опционально), единица измерения (выбор из списка), категория (выбор из списка), описание.
- Убрать текущий inline‑create постоянной ТМЦ (см. модуль 5).

**Зависимости**:
- Модуль 1 (TemporaryItemsAPI) не требуется для создания операции, но нужен для модерации.
- Существующий `CatalogService` для загрузки списков единиц и категорий.

### 3. Корректное чтение истории операций
**Цель**: обеспечить историческую корректность отображения строк операции после approve/merge временных ТМЦ.

**Изменяемые файлы**:
- `apps/operations/services.py` – метод `_present_operation()`
- `templates/operations/detail.html` – отображение строк
- `templates/operations/list.html` – возможно, краткое отображение

**Логика отображения**:

Для каждой строки операции:
1. **Основное имя**: если есть `item_name_snapshot` – использовать его, иначе брать имя из каталога по `item_id`.
2. **SKU**: `item_sku_snapshot` → каталог.
3. **Единица измерения**: `unit_symbol_snapshot` → каталог.
4. **Признак временной ТМЦ**: если есть `temporary_item_id`, показывать badge «Временная ТМЦ».
5. **Разрешение**: если есть `resolved_item_id` и `resolved_item_name`, показывать подпись «Разрешена в ТМЦ …».

**Изменения в `_present_operation()`**:
- Принимать snapshot‑поля из ответа сервера (`line.get('item_name_snapshot')` и т.д.).
- Не полагаться только на `items_index` для строк, у которых есть `temporary_item_id`.
- Добавить в выходной словарь строки поля `is_temporary`, `resolved_item_id`, `resolved_item_name`.

**Тесты**: unit‑тесты, проверяющие приоритет snapshot‑полей.

### 4. Раздел модерации временных ТМЦ
**Цель**: создать отдельный Django‑app для просмотра списка временных ТМЦ, детальной карточки и выполнения действий approve/merge.

**Новый app**: `apps/temporary_items`

**Структура файлов**:
```
apps/temporary_items/
├── __init__.py
├── apps.py
├── urls.py
├── views.py
├── services.py
└── tests.py
templates/temporary_items/
├── list.html
├── detail.html
└── includes/ (опционально)
```

#### 4.1. Маршруты (`urls.py`)
- `GET /temporary-items/` – список с фильтрами (статус, поиск, resolved_item_id)
- `GET /temporary-items/<id>/` – карточка временной ТМЦ
- `POST /temporary-items/<id>/approve/` – approve as item
- `POST /temporary-items/<id>/merge/` – merge to existing item

#### 4.2. Список временных ТМЦ (`views.py`, `list.html`)
- Таблица с колонками: ID, имя, SKU, статус, создана, создал, связанная ТМЦ.
- Фильтры: статус (`active`, `approved_as_item`, `merged_to_item`), поиск по имени/SKU.
- Пагинация через `TemporaryItemsAPI.list_temporary_items()`.

#### 4.3. Карточка временной ТМЦ (`detail.html`)
- Отображение всех полей `TemporaryItemResponse`.
- Блок «История операций» – вызов `TemporaryItemsAPI.list_temporary_item_operations()`.
- Кнопки действий (только для статуса `active`):
  - **Approve as item** – подтверждение, что временная ТМЦ становится новой постоянной ТМЦ.
  - **Merge to existing item** – открытие модального окна с выбором существующей ТМЦ из каталога.

#### 4.4. Интеграция с sidebar
Добавить пункт «Временные ТМЦ» в блок «Номенклатура» для пользователей с `can_manage_catalog`:
```html
{% if user|can_manage_catalog %}
  <div class="section">Номенклатура</div>
  <div class="sidebar-subnav">
    ...
    <a href="{% url 'temporary_items:list' %}">Временные ТМЦ</a>
  </div>
{% endif %}
```

**Зависимости**:
- Модуль 1 (TemporaryItemsAPI)
- Существующий `CatalogService` для выбора целевой ТМЦ при merge.
- Права доступа `can_manage_catalog`.

## Порядок выполнения этапов

1. **Этап 1 – Транспорт и контракты**
   - Создать `temporary_items_api.py` и покрыть тестами.
   - *Результат*: клиент для работы с endpoint‑ами временных ТМЦ.

2. **Этап 2 – Создание операций с временными ТМЦ**
   - Расширить черновик операции, добавить генерацию `client_request_id`.
   - Переработать `_build_create_payload()`.
   - Обновить UI формы операции (добавить временные строки).
   - *Результат*: storekeeper может создать операцию с временной ТМЦ.

3. **Этап 3 – Корректное чтение истории операций**
   - Обновить `OperationPageService._present_operation()`.
   - Обновить `detail.html` для отображения snapshot‑полей и resolved item.
   - *Результат*: карточка операции корректно показывает исторические данные.

4. **Этап 4 – Раздел модерации**
   - Создать app `temporary_items` со списком, карточкой и действиями.
   - Интегрировать в sidebar.
   - *Результат*: chief_storekeeper может просматривать и разрешать временные ТМЦ.

5. **Этап 5 – Устранение inline‑create**
   - Удалить или переработать `OperationItemCreateView`.
   - Обновить UI формы операции.
   - *Результат*: создание постоянной ТМЦ из формы операции больше невозможно.

6. **Этап 6 – Адаптация остатков** (вторая очередь)
   - Отложить, но добавить TODO в код.

## Интеграционные аспекты

### SyncServerClient
- Все запросы к временным ТМЦ должны выполняться через `SyncServerClient` с корректными `acting_user_id` и `acting_site_id`.
- Ошибки обрабатываются через существующие `SyncServerAPIError`.

### Права доступа
- Модерация временных ТМЦ доступна только пользователям с `can_manage_catalog` (root, chief_storekeeper).
- Создание операций с временными ТМЦ доступно всем, кто может создавать операции (storekeeper, chief_storekeeper, root).

### Sidebar
- Пункт «Временные ТМЦ» добавляется только при наличии права `can_manage_catalog`.

### Тестирование
- Unit‑тесты для новых методов API.
- View‑тесты для проверки доступа к модерации.
- Интеграционные тесты на стенде: создание операции с временной ТМЦ → модерация → проверка истории.

## Риски и ограничения

1. **Совместимость с существующими операциями**: изменения в `_present_operation()` не должны сломать отображение старых операций.
2. **Производительность**: добавление snapshot‑полей увеличит размер ответа сервера, но клиент уже получает эти данные.
3. **Безопасность**: необходимо убедиться, что `client_request_id` генерируется на клиенте и не подменяется.
4. **UX**: переход от inline‑create к отдельному разделу модерации может потребовать обучения пользователей.

---

## Дополнительные рекомендации

- Использовать `uuid4` для генерации `client_request_id`.
- Временные строки в черновике хранить с уникальным `client_key` (например, `tmp-<timestamp>-<random>`).
- Для выбора категории и единицы измерения в форме временной ТМЦ использовать те же списки, что и в форме создания постоянной ТМЦ (`CatalogService.list_admin_categories()`, `CatalogService.list_admin_units()`).
- В карточке временной ТМЦ выводить историю операций в виде таблицы с ссылками на соответствующие операции.

---

## Заключение
План описывает полный цикл внедрения временных ТМЦ в Django‑клиент, от транспортного слоя до UI‑модерации. Реализация разбита на логические этапы, каждый из которых может быть выполнен независимо, но с учётом зависимостей. После выполнения этапов 1–4 система будет готова к промышленному использованию сценария временных ТМЦ.
