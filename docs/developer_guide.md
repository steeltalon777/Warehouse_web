# Warehouse_web Developer Guide

## 1. Project Overview

**Warehouse_web** — это Django web-клиент складской системы. Приложение ориентировано на работу со справочными данными склада и предоставляет удобный UI для сотрудников.

Ключевые задачи текущей версии клиента:
- работа со справочниками склада;
- ведение категорий;
- ведение ТМЦ (товарно-материальных ценностей / позиций номенклатуры);
- ведение единиц измерения.

Системный контекст:
- источник истины по данным — **SyncServer** (FastAPI);
- Django в этом проекте играет роль web-клиента, а не центрального backend-ядра бизнес-логики;
- пользователь взаимодействует с интерфейсом в браузере.

---

## 2. Architecture

Архитектура системы построена по модели «UI-клиент + внешний API»:

```text
FastAPI (SyncServer)
        ↓
      HTTP API
        ↓
Django Web Client (Warehouse_web)
        ↓
      Browser
```

Что это означает на практике:
- Django-часть отвечает за шаблонный интерфейс, маршрутизацию страниц и проверку прав в UI;
- интеграция с внешним API выполняется через HTTP-клиенты (`httpx`);
- мастер-данные в долгосрочной архитектуре должны поступать из SyncServer.

---

## 3. Tech Stack

Технологии проекта:
- **Python**;
- **Django** (шаблонный MVC/MVT web-framework);
- **HTML Templates** (Django templates);
- БД: в текущей конфигурации используется **SQLite**, проектная архитектура допускает **PostgreSQL**;
- первичные ключи в доменных моделях каталога — **UUID**.

---

## 4. Project Structure

Основная структура репозитория:

- `config/` — конфигурация Django-проекта.
  - `settings.py` — настройки приложений, БД, шаблонов, статики и интеграционных параметров (включая URL SyncServer).
  - `urls.py` — корневая маршрутизация (подключение `catalog`, `client`, `documents`, `users`, `admin`).
- `apps/` — бизнес-приложения проекта.
  - `catalog/` — справочники склада (категории, единицы, ТМЦ).
  - `users/` — роли и профиль пользователя, вход/выход.
  - `common/` — общие компоненты и функции доступа/разрешений.
  - `integration/` — интеграционный слой и HTTP-клиент для SyncServer.
  - также присутствуют `client/` (dashboard/UI) и `documents/` (заготовка модуля документов).
- `templates/` — общие и прикладные Django-шаблоны.
- `static/` — статические ресурсы проекта.
- `manage.py` — стандартная точка входа Django-команд.

## 5. Catalog Application

Приложение `apps/catalog` — центральный модуль текущей версии системы (справочники).

### `models.py`

Основные модели:

1. **Category**
   - дерево категорий (self-relation через `parent`);
   - поддерживает иерархию (родитель/дети), сортировку, активность;
   - содержит защиту от циклов в `clean()`;
   - использует UUID как PK.

2. **Unit**
   - справочник единиц измерения (`code`, `name`);
   - `code` уникален;
   - использует UUID как PK.

3. **Item**
   - складская позиция / ТМЦ;
   - может быть привязана к категории (`category`, nullable);
   - обязательно привязана к единице измерения (`unit`, `PROTECT`);
   - использует UUID как PK.

### Связи моделей

- `Category (1) -> (N) Item` через `Item.category`;
- `Unit (1) -> (N) Item` через `Item.unit`;
- `Category (self FK)` формирует дерево рубрикатора.

### `views.py`

Реализованы CRUD-экраны для справочников:
- категории: список, создание, редактирование, удаление, дерево;
- единицы: список, создание;
- позиции: список, создание.

Все представления защищены:
- требуется авторизация (`LoginRequiredMixin` или явная проверка);
- дополнительно применяется `can_manage_catalog` из `common.permissions`.

### `forms.py`

`ModelForm`-формы:
- `CategoryForm`;
- `UnitForm`;
- `ItemForm`.

Формы повторяют поля соответствующих доменных моделей и используются в `CreateView/UpdateView`.

### `templates/catalog`

Шаблоны интерфейса каталога:
- домашняя страница каталога;
- списки категорий/единиц/позиций;
- формы создания/редактирования;
- удаление категории;
- визуализация дерева категорий.

## 6. Users Application

`apps/users` отвечает за пользователя, роли и аутентификацию.

Ролевая модель (enum `Role`):
- **Root** (`root`);
- **Главный кладовщик** (`chief_storekeeper`);
- **Кладовщик** (`storekeeper`).

Профиль пользователя (`UserProfile`) связан с Django user через OneToOne и хранит:
- роль;
- привязку к площадке/складу (`Site`);
- активность профиля.

Маршруты `users` предоставляют login/logout на базе стандартных `django.contrib.auth.views`.

Проверка прав в модуле каталога выполняется через функции из `common.permissions`.

---

## 7. Common Application

`apps/common` содержит переиспользуемую инфраструктуру:

- `permissions.py`:
  - проверки ролей (`is_root`, `is_chief_storekeeper`, `is_storekeeper`);
  - функциональные права (`can_manage_catalog`, `can_use_client`);
  - утилиты проверки пользователя/профиля.

- `services/syncserver_client.py`:
  - общий HTTP-клиент для запросов в SyncServer (пример: получение категорий).

Этот слой нужен для централизации кросс-модульной логики, чтобы не дублировать её в каждом приложении.

---

## 8. Integration

`apps/integration` — пространство для интеграции с SyncServer API.

Текущее состояние:
- реализован базовый HTTP-клиент `SyncServerClient` на `httpx`;
- URL API берётся из `settings.SYNCSERVER_API_URL`;
- пока покрыт базовый сценарий чтения категорий.

Целевое развитие:
- расширить клиент до полноценного gateway-слоя;
- добавить типизированные методы по доменам (каталог, документы, остатки, операции);
- унифицировать обработку ошибок и retry-поведение.

## 9. Templates

Базовая структура шаблонов:

- `templates/base.html` — общий layout;
- `templates/includes/navbar.html` — верхняя панель;
- `templates/includes/sidebar.html` — боковая навигация (пункты зависят от роли/статуса пользователя);
- `templates/catalog/*` — экраны модуля каталогов;
- `templates/client/dashboard.html` — главная страница клиентской части;
- `templates/registration/login.html` — форма входа.

## 10. Development

Базовый запуск проекта в dev-режиме:

```bash
python manage.py migrate
python manage.py runserver
```

Дополнительно обычно используют:

```bash
python manage.py createsuperuser
python manage.py test
```

## 11. Future Development

Приоритетные направления развития системы:
- модуль **операций склада**;
- модуль **остатков**;
- модуль **документов**;
- углублённая интеграция с **SyncServer API** (двусторонний обмен и синхронизация).

Рекомендуемый вектор эволюции архитектуры:
- сохранить Django как UI/BFF-подобный клиентский слой;
- бизнес-правила и согласованность данных концентрировать в SyncServer;
- в Django развивать UX, role-based доступ и операционные сценарии пользователей.
